## 1. Requirements Understanding

### 1.1 Problem Statement

북랩의 현재 사이트는 출판 패키지, 제작 설비, 가격, 유통, 판매 확인 등 의사결정에 필요한 자산을 갖고 있다. 하지만 첫 방문자가 보기에는 정보가 기능/페이지 단위로 흩어져 있고, "내 상황에서 무엇을 선택해야 하는가"와 "상담을 지금 시작해도 되는가"에 대한 답이 약하다.

좋은땅은 메인에서 상담, 출판 과정, 저자 후기, 도서 제작 현황, 인세/마케팅 신뢰 장치를 반복 노출해 예비 저자의 불안을 줄인다. 북랩 리뉴얼은 이 전환 구조를 벤치마킹하되, 북랩만의 강점인 패키지 가격 투명성, POD 자체 제작, POS 판매 확인, 다양한 목적별 상품을 핵심 차별화로 유지해야 한다.

### 1.2 Confirmed Assumptions

- 1차 리뉴얼은 전체 사이트 재구축보다 홈과 핵심 전환 경로 개선을 우선한다.
- 좋은땅을 그대로 복제하지 않고 상담 전환 구조와 신뢰 배치 방식을 벤치마킹한다.
- 기존 북랩의 출판 패키지 정보와 가격/사양 구조는 유지하되, 진입 UX를 단순화한다.
- 현재 기술 스택과 CMS 구조는 아직 미확정이므로 Design 단계에서 구현 옵션을 나눈다.
- 실제 상담 접수, 원고 전송, CRM 연동은 MVP에서 기존 시스템 연결 또는 임시 접수 방식으로 처리할 수 있다.

- 신규 CRM 구축
- 앱 개발
- 브랜드 로고 전면 교체
- 모든 하위 페이지의 동시 리뉴얼

## 6. UX Requirements

### 6.1 Page Flow

1. Hero: 핵심 메시지, 신뢰 수치, 상담/추천 CTA
2. Purpose selector: 저자 목적별 진입
3. Package recommendation: 4문항 진단 또는 대표 패키지 비교
4. Trust proof: 출간종수, POD센터, POS 판매 확인, 세종도서
5. Publishing journey: 6단계 과정
6. Published books / testimonials: 실제 사례
7. FAQ: 비용/제작/유통/정산 불안 해소
8. Final CTA: 상담 신청, 원고 전송, 전화/카카오톡 안내

### 6.2 UX Principles

- 첫 화면은 설명보다 판단을 돕는다.
- 패키지명보다 사용자의 목적을 먼저 묻는다.
- 상담 CTA는 반복하되, 섹션 맥락에 맞춰 라벨을 다르게 쓴다.
- 북랩의 차별점은 감성 문구보다 검증 가능한 수치와 기능으로 보여준다.
- 모바일에서는 한 화면에 하나의 주요 판단만 제공한다.

## 8. Technical Planning Assumptions

현재 저장소에는 애플리케이션 코드가 없고 `docs` 문서만 생성된 상태다. 따라서 기술 계획은 Design 단계에서 다음 세 가지 구현 옵션으로 나눠 확정한다.

| Option | 설명 | 적합한 상황 |
|---|---|---|
| A. Static Renewal Prototype | HTML/CSS/JS 또는 정적 프론트엔드로 홈 리뉴얼 시안 구현 | 빠른 검토, 기존 시스템과 분리된 목업 필요 |
| B. Existing CMS Integration | 기존 북랩 사이트/CMS 구조에 맞춰 홈 템플릿과 콘텐츠 블록만 개선 | 현 사이트 유지보수가 우선이고 배포 제약이 클 때 |
| C. Modern Frontend Rebuild | Next.js 등으로 홈/랜딩/상담 흐름을 재구축하고 기존 시스템과 연동 | 장기 리뉴얼과 성능/운영 개선까지 고려할 때 |

Plan 단계의 추천은 `A -> B/C 결정`이다. 먼저 전환형 홈과 패키지 추천 UX를 정적 프로토타입으로 검증한 뒤, 실제 배포 구조를 선택하는 편이 리스크가 낮다.

## 10. Risks and Mitigations

| Risk | Severity | Mitigation |
|---|---|---|
| 기존 CMS 구조 때문에 원하는 UI 구현이 어려울 수 있음 | High | Design 단계에서 정적 프로토타입, CMS 통합, 신규 프론트엔드 옵션을 비교한다. |
| 가격/패키지 정보가 자주 바뀌면 유지보수 부담이 커짐 | Medium | 패키지 데이터를 별도 객체/콘텐츠 소스로 분리하는 설계를 검토한다. |
| 좋은땅과 유사한 화면 구성으로 보일 수 있음 | Medium | 북랩 차별 자산인 패키지 투명성, POD센터, POS 확인을 핵심 섹션에 둔다. |
| 상담 폼 항목이 많아 제출률이 낮아질 수 있음 | Medium | 1차 폼은 최소 입력으로 제한하고 상세 정보는 후속 상담에서 수집한다. |
| 후기 콘텐츠가 부족할 수 있음 | Low | MVP는 출간 도서, 수상/선정, 제작 역량으로 신뢰를 보완한다. |

## 12. Open Questions for Design Phase

| Question | Why It Matters | Default Assumption |
|---|---|---|
| 현재 북랩 사이트는 어떤 CMS/언어/배포 구조인가? | 실제 구현 방식 결정 | 정적 프로토타입 우선 |
| 상담 신청은 기존 폼으로 연결할지 새 폼을 만들지? | 전환 추적과 운영 흐름 결정 | 기존 접수 방식 연결 |
| 카카오톡/전화/이메일/원고 전송 중 1순위 CTA는 무엇인가? | 모바일 고정 CTA 설계 | 상담 신청 1순위, 전화/카카오 보조 |
| 패키지 가격은 실가격을 노출할 수 있는가? | 가격 투명성 메시지 강도 | 기준 가격 또는 "예상 비용 확인" CTA 병행 |
| 후기/출간 사례 콘텐츠가 사용 가능한가? | 신뢰 섹션 완성도 | 출간 도서/수상 이력으로 MVP 대체 |
